# Agent Update: Deployment Build Testing for Vercel and Local

## Purpose
Ensure the agent always tests both Vercel and local builds for deployment readiness, with special handling for Babel config and Node/Next.js version mismatches.

## Instructions

1. **Babel Config Handling**
   - Always check for the presence of `babel.config.js`, `babel.config.cjs`, and `babel.config.mjs`.
   - For Next.js 13+ (especially with Turbopack), only `babel.config.js` (CommonJS) is supported.
   - If `package.json` contains `"type": "module"`, warn the user and recommend removing it for Next.js compatibility.
   - Remove or rename any `.mjs` or `.cjs` Babel configs before deployment.

2. **Version Checks**
   - Check that Node.js and Next.js versions match Vercel's build environment (use `.vercel/project.json` or Vercel dashboard for reference).
   - Warn if local Node.js version differs from Vercel's default.

3. **Build Testing**
   - Always run `npm run build` locally and capture errors.
   - If local build passes, simulate Vercel build by running with the same Node.js version as Vercel (use `nvm` or Docker if needed).
   - If Vercel build fails but local passes, check for ESM/CommonJS config issues, missing dependencies, or version mismatches.

4. **Reporting**
   - Summarize any build errors with actionable fixes (e.g., Babel config, Node version, missing packages).
   - Suggest exact file/config changes to resolve deployment blockers.

5. **PR/Commit Guidance**
   - Ensure all build/deployment fixes are committed and pushed before PR creation.
   - Add a checklist to PRs for deployment readiness (local build, Vercel build, Babel config, Node version).

---

**Example Checklist for PRs:**
- [ ] Local `npm run build` passes
- [ ] No `.mjs` or `.cjs` Babel config files
- [ ] `babel.config.js` present and CommonJS
- [ ] No `"type": "module"` in `package.json`
- [ ] Node.js version matches Vercel
- [ ] Vercel build passes

---

**Note:**
This agent update is to prevent repeated deployment failures due to Babel config or version mismatches. Always validate both local and Vercel builds before marking a deployment as ready.
